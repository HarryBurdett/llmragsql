Opera Database Locking Rules — Complete Reference

Generated: 09/04/2026 09:50

These rules are MANDATORY for every application that writes to Opera SE or Opera 3 databases. Failure to follow them will cause data corruption, deadlocks, and blocked users.

Golden Rules

RULE 1: Every SELECT from an Opera table MUST include WITH (NOLOCK)
Without it, SQL Server takes shared locks that block Opera users from editing records. No exceptions.
RULE 2: Every UPDATE must include WITH (ROWLOCK)
Prevents table-level locks. Other users can update different rows simultaneously.
RULE 3: Every write method must be wrapped in deadlock retry
3 attempts, exponential backoff (0.1s, 0.5s, 1.5s). SQL error 1205 = deadlock victim, retry is safe.
RULE 4: Sequence allocation uses UPDLOCK, ROWLOCK on the SELECT
Prevents two processes getting the same journal/entry number. Lock the row BEFORE reading it.
RULE 5: Application-level import lock per bank account
Prevents concurrent imports posting to the same bank. Acquired before any cashbook write, released in finally block.

Complete Write Pattern

1. VALIDATE — SELECT WITH (NOLOCK) to check accounts exist, not dormant, period open
2. ACQUIRE import lock (if cashbook posting)
3. BEGIN TRANSACTION (inside deadlock retry wrapper)
4. ALLOCATE sequences:
   - Journal: SELECT np_nexjrnl FROM nparm WITH (UPDLOCK, ROWLOCK)
              UPDATE nparm WITH (ROWLOCK) SET np_nexjrnl = np_nexjrnl + 1
   - Entry:  SELECT ay_entry FROM atype WITH (UPDLOCK, ROWLOCK) WHERE ay_code = cbtype
              UPDATE atype WITH (ROWLOCK) SET ay_entry = ay_entry + 1
   - Row ID: SELECT nextid FROM nextid WITH (UPDLOCK, ROWLOCK) WHERE tablename = 'table'
              UPDATE nextid WITH (ROWLOCK) SET nextid = nextid + 1
5. INSERT new records (aentry, atran, stran/ptran, ntran, anoml)
6. UPDATE balances WITH (ROWLOCK):
   - nbank.nk_curbal (bank balance, in pence)
   - sname.sn_currbal / pname.pn_currbal (customer/supplier balance)
   - nacnt period/YTD balances
   - nhist nominal history
7. COMMIT TRANSACTION
8. RELEASE import lock (in finally block — always executes)

Opera 3 (FoxPro) Locking Rules

Opera 3 uses FoxPro DBF files, not SQL Server. Different locking mechanism:

  • Record lock: RLOCK() before updating a single record, UNLOCK immediately after
  • File lock: FLOCK() only when absolutely necessary (e.g. appending), release immediately
  • Shared access: DBF files on SMB share — multiple readers, single writer per record
  • Write Agent: All Opera 3 writes must go through the Write Agent service on the server (proper FoxPro locking)
  • Never hold locks: Open file → lock record → write → unlock → close. No held locks between operations.
