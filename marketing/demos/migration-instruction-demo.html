Architecture Overview

The system follows a clear provider/consumer pattern. Core provides shared services. Apps are self-contained modules that consume those services.

Key Principle

Each app owns its own API routes, business logic, frontend pages, per-company data, and app-specific settings. Everything else comes from core.

1

Step 1: Verify Self-Containment

Before migrating, confirm the app has NO dependencies outside its own structure — except for core services.

  • All API routes are in apps/{app}/api/
  • All app-specific data is in data/{company}/{app}/
  • App settings are stored per-company in the app's data folder
  • No hardcoded paths to other apps
  • No imports from other apps (apps import from core only, NEVER from each other)
  • Frontend pages reference only /api/ endpoints (not internal module paths)

Why this matters

An app that references another app cannot be deployed independently. Cross-app dependencies create coupling that breaks the migration model.

2

Step 2: Identify Core Dependencies

List every core service the app uses. This becomes the app's dependency manifest.

Example: bank_reconcile

dependencies: - sql_connector # Query Opera cashbook, ptran, stran, etc. - email_storage # Email scanning for bank statements - email_sync_manager # IMAP provider access - config # System configuration - current_company # Company context - opera_sql_import # Post transactions to Opera SE - opera3_foxpro_import # Post transactions to Opera 3 - opera_config # Period posting, control accounts - import_lock # Bank-level locking

Other apps are simpler

balance_check needs only sql_connector. dashboards also needs only sql_connector. The manifest makes the dependency surface explicit.

3

Step 3: Register in New System

In the target system, the app registers itself via FastAPI's router pattern.

# Each app provides a router from apps.bank_reconcile.api.routes import router as bank_rec_router # The new system's main.py: app.include_router(bank_rec_router) # Core provides services via dependency injection or globals sync: from apps.core.state import _sync_from_main # OR via a service locator: from apps.core.services import get_sql_connector, get_email_storage

Option A: State Sync

Core syncs global state (sql_connector, etc.) to a shared module that apps import from.

Option B: Service Locator

Apps call core service functions that return the current connector, storage, etc. at runtime.

4

Step 4: Migrate Data

Per-company data moves with the app. The folder structure is identical on source and target.

Source System

data/{company}/bank_reconcile/ bank_aliases.db bank_patterns.db import_locks.db pdf_extraction_cache.db

Target System

data/{company}/bank_reconcile/ bank_aliases.db ← same bank_patterns.db ← same import_locks.db ← same pdf_extraction_cache.db ← same

Automatic Path Resolution

The get_company_db_path() function handles path resolution automatically based on company context. No path changes needed when moving between systems.

6

Step 6: Verify

After migration, run through the verification checklist to confirm everything works.

  • All API endpoints return 200
  • Multi-company switching works
  • Data isolation between companies verified
  • Frontend pages load and function
  • Opera SE endpoints work
  • Opera 3 endpoints work (if applicable)
  • Settings persist across restarts
  • No references to other apps (only core)

Core Settings Checklist

When setting up a new installation, also verify: config.ini updated for target environment, companies/*.json has correct database names, SQL connection works, email connection works, and initial company context creates data directories.

The 7 Golden Rules

Every migration must honour these rules. Violations break the architecture.

  1. Apps import from core — NEVER from each other
  2. Core provides services — apps consume them
  3. Each app's data is self-contained in data/{company}/{app}/
  4. System-wide config stays with core
  5. App-specific config stays with the app
  6. No hardcoded paths, account codes, or company-specific values
  7. Both Opera SE and Opera 3 must work
✓
